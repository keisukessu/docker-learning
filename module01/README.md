## 1. 仮想マシン（VM）とコンテナの違い

### 仮想マシン（VM）のイメージ

```
┌─────────────────────────────────────────┐
│              物理サーバー                 │
│  ┌──────────────────────────────────┐   │
│  │       ハイパーバイザー             │   │
│  │  ┌──────────┐  ┌──────────┐     │   │
│  │  │  VM 1    │  │  VM 2    │     │   │
│  │  │  OS全体  │  │  OS全体  │     │   │
│  │  │  (数GB)  │  │  (数GB)  │     │   │
│  │  │  アプリ  │  │  アプリ  │     │   │
│  │  └──────────┘  └──────────┘     │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**特徴:**
- OS 全体を仮想化するため重い（数GB）
- 起動に数分かかる
- 完全に独立しているので安全

### Docker コンテナのイメージ

```
┌─────────────────────────────────────────┐
│              物理サーバー                 │
│              OS（Linux/Windows）          │
│  ┌──────────────────────────────────┐   │
│  │       Docker Engine              │   │
│  │  ┌──────┐  ┌──────┐  ┌──────┐  │   │
│  │  │コンテナ│  │コンテナ│  │コンテナ│  │   │
│  │  │アプリ │  │アプリ │  │アプリ │  │   │
│  │  │(数MB) │  │(数MB) │  │(数MB) │  │   │
│  │  └──────┘  └──────┘  └──────┘  │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**特徴:**
- ホスト OS のカーネルを共有するので軽い（数MB〜数百MB）
- 起動が数秒で完了
- 環境をコードで管理できる

### 比較表

| 項目 | 仮想マシン | コンテナ |
|------|-----------|---------|
| サイズ | 数GB | 数MB〜数百MB |
| 起動時間 | 数分 | 数秒 |
| OS | 各VMに完全なOS | ホストOSを共有 |
| 独立性 | 完全に独立 | プロセスレベルで独立 |
| 用途 | 異なるOSが必要な場合 | アプリケーションの配布 |

## 2. Docker のアーキテクチャ

```
┌─────────────────────────────────────────────────┐
│                  Docker Hub                      │
│          (クラウド上のイメージ置き場)               │
│  [nginx] [ubuntu] [python] [mysql] ...           │
└──────────────────┬──────────────────────────────┘
                   │ docker pull / docker push
┌──────────────────▼──────────────────────────────┐
│              あなたのPC                           │
│  ┌─────────────────────────────────────────┐    │
│  │           Docker Engine                  │    │
│  │                                          │    │
│  │  Images（設計図）                         │    │
│  │  ┌──────┐ ┌──────┐ ┌──────┐            │    │
│  │  │nginx │ │ubuntu│ │python│            │    │
│  │  └──────┘ └──────┘ └──────┘            │    │
│  │       docker run ↓                      │    │
│  │  Containers（動いているもの）              │    │
│  │  ┌──────┐ ┌──────┐                     │    │
│  │  │cont1 │ │cont2 │                     │    │
│  │  └──────┘ └──────┘                     │    │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
```

### 主要な概念

- **Image（イメージ）**: アプリの設計図。読み取り専用。
- **Container（コンテナ）**: イメージを実行したもの。動いているプロセス。
- **Registry（レジストリ）**: イメージを保存・配布する場所。Docker Hub が代表的。

## 3. なぜ現場で Docker が使われるのか

### 「自分のPCでは動くのに...」問題を解決

```
開発者A: "俺のPCでは動いてるよ？"
開発者B: "私のPCじゃ動かないんだけど..."

↓ Docker を使うと

全員: "コンテナで動かすから環境の差がない！"
```

### 現場でのメリット

1. **環境の統一**: チーム全員が同じ環境で開発できる
2. **素早いセットアップ**: 新しいメンバーが `docker compose up` 一発で環境構築完了
3. **本番環境と同じ**: ローカルで動いたものがそのまま本番でも動く
4. **複数バージョン共存**: Node.js 18 と Node.js 20 を同時に使える

### 就職・転職での価値

求人でよく見るスキル要件：
- ✅ Docker / Docker Compose
- ✅ コンテナ化
- ✅ CI/CD（GitHub Actions など）

Docker を使いこなせると、エンジニア転職の市場価値が上がります。

## ハンズオン: 最初のコンテナを起動してみよう

```bash
docker run hello-world
```

### 出力を読み解く

```
Hello from Docker!                          ← コンテナが動いた証拠
This message shows that your installation appears to be working correctly.

To generate this message, Docker took the following steps:
 1. The Docker client contacted the Docker daemon.      ← クライアントがデーモンに接続
 2. The Docker daemon pulled the "hello-world" image    ← Docker Hub からイメージを取得
    from the Docker Hub. (amd64)
 3. The Docker daemon created a new container from      ← イメージからコンテナを作成
    that image which runs the executable that produces
    the output you are currently reading.
 4. The Docker daemon streamed that output to the       ← 出力をターミナルに表示
    Docker client, which sent it to your terminal.
```
